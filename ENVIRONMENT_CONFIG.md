# Environment Configuration Guide

Complete guide for configuring environment variables for the Project Management System.

## Overview

Environment variables are used to:
- Store sensitive information (passwords, tokens, secrets)
- Adjust behavior for different environments (dev, staging, production)
- Avoid hardcoding configuration

## Backend Environment Variables

### Location
`backend/.env`

### Variables

#### Database Configuration
```
DATABASE_URL=postgresql://username:password@localhost:5432/project_management
```
- Format: `postgresql://user:password@host:port/database_name`
- For Docker Compose: `postgresql://postgres:postgres@postgres:5432/project_management`
- For Railway: Auto-generated by Railway

**Required**: Yes
**Default**: None
**Example**: `postgresql://project_user:secure_pass@localhost:5432/project_management`

#### JWT Configuration
```
JWT_SECRET=your_very_secure_random_secret_here_minimum_32_chars
```
- Used to sign and verify JWT tokens
- Must be kept secret and never committed to git
- Generate strong secret: `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`

**Required**: Yes
**Default**: None
**Min Length**: 32 characters
**Example**: `a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6`

```
JWT_EXPIRATION=7d
```
- Token expiration time
- Format: `<number><unit>` where unit is `d` (day), `h` (hour), `m` (minute), `s` (second)

**Required**: No
**Default**: `7d`
**Examples**: `7d`, `24h`, `1h`, `30m`

#### Server Configuration
```
PORT=5000
```
- Port on which the backend server runs

**Required**: No
**Default**: `5000`
**Range**: 1024-65535

```
NODE_ENV=development
```
- Environment mode

**Required**: No
**Default**: `development`
**Valid Values**: `development`, `production`, `staging`, `test`

#### CORS Configuration
```
CORS_ORIGIN=http://localhost:5173
```
- Allowed frontend origin for CORS
- For multiple origins, separate with comma (not recommended in production)
- Use `*` only for development, never in production

**Required**: No
**Default**: `http://localhost:5173`
**Examples**:
- Development: `http://localhost:5173`
- Production: `https://your-app-domain.com`
- Multiple: `http://localhost:5173,https://your-domain.com`

### Complete Backend .env Example

```env
# Database
DATABASE_URL=postgresql://project_user:secure_password@localhost:5432/project_management

# JWT
JWT_SECRET=a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2c3d4e5f6a1b2
JWT_EXPIRATION=7d

# Server
PORT=5000
NODE_ENV=development
CORS_ORIGIN=http://localhost:5173
```

## Frontend Environment Variables

### Location
`frontend/.env`

### Variables

#### API Configuration
```
VITE_API_URL=http://localhost:5000
```
- Base URL for API requests
- Used by axios to construct endpoints
- Automatically used in API calls via `/api` path

**Required**: No
**Default**: `http://localhost:5000`
**Examples**:
- Development: `http://localhost:5000`
- Production: `https://api.your-domain.com`

#### Optional Variables
```
VITE_API_TIMEOUT=30000
```
- API request timeout in milliseconds

**Required**: No
**Default**: `30000`

```
VITE_DEBUG=false
```
- Enable debug logging

**Required**: No
**Default**: `false`
**Valid Values**: `true`, `false`

### Complete Frontend .env Example

```env
# API Configuration
VITE_API_URL=http://localhost:5000

# Optional
VITE_API_TIMEOUT=30000
VITE_DEBUG=false
```

## Docker Environment Variables

### Docker Compose (.env file in root directory)

If using `docker-compose.yml`:

```env
# PostgreSQL
POSTGRES_USER=postgres
POSTGRES_PASSWORD=postgres
POSTGRES_DB=project_management

# Backend
BACKEND_PORT=5000
NODE_ENV=development

# Frontend
FRONTEND_PORT=5173
VITE_API_URL=http://localhost:5000
```

## Railway Deployment Environment Variables

### Backend Service Variables

Set these in Railway dashboard for backend:

```
JWT_SECRET=<your_secure_secret>
JWT_EXPIRATION=7d
NODE_ENV=production
PORT=5000
CORS_ORIGIN=https://your-frontend-domain.up.railway.app
```

**Auto-provided by Railway**:
- `DATABASE_URL` (from PostgreSQL service connection)

### Frontend Service Variables

Set these in Railway dashboard for frontend:

```
VITE_API_URL=https://your-backend-domain.up.railway.app
```

## Development Setup

### Quick Setup

```bash
# Backend
cd backend
cp .env.example .env
# Edit .env with your local configuration
nano .env

# Frontend
cd frontend
cp .env.example .env
# Keep defaults or edit as needed
```

### Using Docker Compose

Docker Compose automatically handles environment variables:

```bash
docker-compose up
```

All services get pre-configured environment variables.

## Production Setup

### Railway Setup

1. Go to Railway dashboard
2. Click on service
3. Go to "Variables" tab
4. Add each variable one by one
5. Variables are encrypted and secure

### Manual VPS/Server Setup

```bash
# Create .env file
touch backend/.env

# Add variables securely
cat > backend/.env << EOF
DATABASE_URL=postgresql://...
JWT_SECRET=...
NODE_ENV=production
PORT=5000
CORS_ORIGIN=https://your-domain.com
EOF

# Set permissions
chmod 600 backend/.env

# Never commit .env to git
echo ".env" >> backend/.gitignore
```

## Validation Checklist

Before deploying, verify:

- [ ] `JWT_SECRET` is strong (minimum 32 characters, random)
- [ ] `DATABASE_URL` points to correct database
- [ ] `CORS_ORIGIN` matches frontend URL exactly
- [ ] `NODE_ENV` is set appropriately (development/production)
- [ ] `.env` file is in `.gitignore`
- [ ] All required variables are set
- [ ] No hardcoded secrets in code

## Security Best Practices

### Do's
✅ Use strong random secrets for `JWT_SECRET`
✅ Keep `.env` file locally only
✅ Use environment variables for all sensitive data
✅ Rotate `JWT_SECRET` periodically
✅ Use HTTPS URLs in production
✅ Keep `.env` in `.gitignore`

### Don'ts
❌ Never commit `.env` to git
❌ Never share `.env` file
❌ Don't use simple secrets (like "secret123")
❌ Don't hardcode configuration in code
❌ Don't use same secret across environments
❌ Don't share database URL in public channels

## Troubleshooting

### Issue: "Cannot connect to database"

**Check**:
```bash
# Verify DATABASE_URL format
echo $DATABASE_URL

# Test connection
psql $DATABASE_URL

# Check if PostgreSQL is running
pg_isready
```

**Solution**:
- Verify host, port, username, password
- Ensure database exists
- Check PostgreSQL service is running

### Issue: "Invalid token" errors

**Check**:
```bash
# Verify JWT_SECRET matches between services
echo $JWT_SECRET
```

**Solution**:
- Ensure same `JWT_SECRET` on backend
- Regenerate token after changing secret
- Check JWT hasn't expired

### Issue: CORS errors in browser

**Check**:
```bash
# Verify CORS_ORIGIN
echo $CORS_ORIGIN
```

**Solution**:
- Ensure `CORS_ORIGIN` matches frontend URL exactly
- Include protocol (http/https)
- Restart backend after changing

### Issue: API requests timeout

**Check**:
```bash
# Verify API_URL
echo $VITE_API_URL

# Test backend is running
curl $VITE_API_URL/health
```

**Solution**:
- Check backend is running
- Verify network connectivity
- Increase `VITE_API_TIMEOUT` if needed

## Environment Variable Precedence

### Backend (Node.js)
1. System environment variables
2. `.env` file
3. Hardcoded defaults in code
4. Undefined

### Frontend (Vite)
1. System environment variables (with `VITE_` prefix)
2. `.env` file
3. Hardcoded defaults
4. Undefined

## Regenerating Secrets

If you need to regenerate `JWT_SECRET`:

```bash
# Generate new secret
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"

# Update .env
# Restart backend
# All users need to re-login
```

## Multiple Environments

### Setup Structure

```
backend/
├── .env.example          # Template
├── .env                  # Development (local)
├── .env.production       # Production secrets (not in git)
├── .env.staging          # Staging secrets (not in git)
```

### Load Appropriate .env

```bash
# Development
npm run dev

# Production
NODE_ENV=production npm start

# Staging
NODE_ENV=staging npm start
```
